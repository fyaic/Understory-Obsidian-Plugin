# Understory Agent Skill

Understory is the local search and relation layer for one Obsidian vault. The MCP server finds relevant notes, relation metadata, snippets, and nearby context; the agent turns those results into a useful answer, reading path, or knowledge map.

This Skill is designed to travel with the plugin + MCP setup. Do not hardcode an author machine path into this file. The vault path and MCP server key come from the setup pack generated inside the user's current vault.

---

## When To Use

Use this Skill when the user asks to:

- search, cite, summarize, or inspect the current Obsidian vault
- find related notes for a topic, product question, project, architecture decision, or research direction
- turn Understory results into a reusable related-notes index or knowledge map
- use the vault as long-term local context for an ongoing project
- inspect or maintain Understory relation metadata after user confirmation

Do not use this Skill for a simple filesystem read of a known local file unless vault search or relation context is useful.

---

## Core Principle

Understory is the retrieval engine. The agent is the curator.

Do not stop at a raw list of notes. Upgrade search results into business judgment:

- group notes by product module, mechanism, decision area, risk, or research thread
- explain why each note matters to the current question
- classify notes as required reading, optional background, needs verification, or research gap
- identify what the vault already knows and what is still missing
- propose the next concrete action

## MCP First

Before reading vault files directly, call the matching Understory MCP server for scoped retrieval.

Recommended read tools:

| Tool | Use |
| --- | --- |
| `understory_status` | Confirm vault identity and local engine status. |
| `understory_get_capabilities` | Inspect read/write boundaries and privacy defaults. |
| `understory_search` | Search notes with local keyword and relation-graph signals. |
| `understory_get_context` | Build a scoped context package from a query or known note path. |
| `understory_get_note_brief` | Read a concise brief for one note. |
| `understory_get_relations` | Inspect known relations for one note. |
| `understory_graph_summary` | Get vault-level relation, conflict, and index summary counts. |

Write-capable tools such as `understory_accept_relation`, `understory_reject_relation`, `understory_insert_relation`, and `understory_refresh_relations` change local vault state. Explain the planned local change and ask for user confirmation before using them.

The server is normally bound to one vault by MCP config. Do not pass another `vaultPath` to the same server unless the user explicitly asks for advanced troubleshooting.

## Knowledge Map Workflow

Use this workflow when the user asks for research, positioning, competitive context, project onboarding, related notes, or vault knowledge assets.

### 1. Frame The Business Question

Before searching, identify:

- the exact question to answer
- the target reader, such as PM, engineer, architect, researcher, founder, or operator
- the intended output format, such as chat answer, markdown note, related-notes index, or handoff brief
- the destination if the user wants a file, using a vault-relative path or a user-provided folder

If unclear, make the smallest reasonable assumption and state it.

### 2. Design A Query Matrix

Use 3-5 searches that cover different angles instead of one broad query.

Example matrix:

| Angle | Example query |
| --- | --- |
| Core positioning | `agent memory model local knowledge graph` |
| User identity | `per-user memory session routing user-specific context` |
| Collaboration semantics | `shared agent team member handoff context` |
| Governance | `permission audit write safety local vault` |
| Bilingual synonyms | `成员化 Agent 分用户记忆 知识地图` |

Use `limit` around 10-15 for broad exploration, then narrow with follow-up context reads.

### 3. Read Key Context

For promising results:

- call `understory_get_context` by query or note path
- use `understory_get_note_brief` when you need a fast note-level summary
- inspect `understory_get_relations` when relation structure matters
- read full files only when snippets and relation metadata are insufficient

Prefer notes that appear across multiple query angles, have strong relation context, or map directly to the user's business question.

### 4. Curate And Classify

Deduplicate all candidates and group them by business module. Common groupings:

- product positioning and value proposition
- user identity, permissions, and session routing
- memory architecture and context handoff
- collaboration workflows and team semantics
- governance, safety, audit, and write boundaries
- indirect background and open research gaps

For each important note, state:

- which subproblem it informs
- what mechanism, data model, claim, or product meaning it contributes
- whether it is directly reusable, indirectly useful, background only, or needs verification
- the next action: read, cite, compare, validate, update, or ignore

### 5. Produce A Durable Output

For a markdown deliverable, use `templates/knowledge-map.md` when available. A good output includes:

1. title and metadata
2. reading guide and rating legend
3. key knowledge assets grouped by business module
4. missing knowledge and next research actions
5. recommended reading paths by role
6. full index sorted by business value
7. reproduction notes: query matrix, MCP tools used, and update instructions

When the selected Skill mode is Query-only, return the knowledge map in the conversation unless the user explicitly asks for and confirms a local file write.

## Write Safety

- Default to read-only retrieval and analysis.
- Do not scan the entire vault by default.
- Do not return full note bodies unless the user asks and it is necessary.
- Ask before creating files, inserting links, accepting/rejecting relations, refreshing relation blocks, or writing memory updates.
- Never send vault text, API keys, prompts, embeddings, logs, or webhook URLs to Bondie Labs.

## Path Hygiene

Use placeholders in reusable instructions:

- `<VAULT_ROOT>` for the current vault root
- `<ENGINE_DIR>` for the local `understory-graphify-engine` folder
- `<OUTPUT_DIR>` for the user-selected vault-relative output folder

Accept vault-specific paths only when they are generated by the plugin setup pack for that user's machine. Do not bake examples such as an author username, a personal project folder, or a private vault name into reusable docs.

Example MCP config shape:

```json
{
  "mcpServers": {
    "understory-current-vault": {
      "command": "node",
      "args": [
        "<VAULT_ROOT>/.understory/agent/understory-mcp-server.js",
        "--vault",
        "<VAULT_ROOT>",
        "--engine-dir",
        "<ENGINE_DIR>",
        "--python-path",
        "python"
      ]
    }
  }
}
```

## Quality Bar

Reject and rewrite the answer if it:

- only lists search hits
- only says a note matched a keyword
- mixes model guesses with vault-backed facts without labeling the difference
- lacks business grouping for a broad research question
- lacks knowledge gaps or next actions
- lacks recommended reading paths for a broad knowledge map
- includes hardcoded local paths from another user's machine

One-line boundary:

> Understory finds the notes; this Skill turns them into an actionable knowledge map.
